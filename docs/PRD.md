# 助教招聘系统 PRD

## 1. 文档信息

- 产品名称：北京邮电大学国际学院助教招聘系统
- 当前阶段：课程项目 / 原型系统
- 文档版本：v1.0
- 文档范围：基于当前代码库已实现能力整理，重点描述现有产品功能、用户流程、边界和后续演进方向。

## 2. 产品背景

高校课程助教招聘通常涉及岗位发布、学生申请、简历筛选、面试沟通、录用分配和工作量管理。传统流程容易分散在群聊、邮件或表格中，存在信息不集中、筛选效率低、沟通记录难追踪、录用工作量不透明等问题。

本系统提供一个轻量级 Web 平台，让应聘者、课程组织者和管理员在同一系统内完成助教招聘闭环，并通过论坛、私信、简历管理、匹配推荐和智能小助手提升申请与筛选效率。

## 3. 产品定位

本项目定位为面向教学场景的助教招聘管理原型系统。

系统当前适合：

- 课程项目演示与本地部署。
- 小规模助教岗位发布和申请管理。
- 展示多角色 Web 系统、JSON 文件持久化、Servlet/JSP 页面流程、基础 AI 接入与额度管理能力。

系统当前不定位为生产级招聘平台。若投入真实业务使用，需要补充数据库、权限审计、HTTPS、密钥管理、备份恢复、并发控制和更完善的隐私合规机制。

## 4. 目标用户

### 4.1 应聘者 TA

TA 是希望申请课程助教、监考或活动支持岗位的学生。

核心诉求：

- 快速注册并维护个人资料和技能标签。
- 上传、编辑和下载简历。
- 浏览岗位并按技能、类型、关键词筛选。
- 提交申请并查看申请状态。
- 接收面试安排，确认、拒绝或申请改期。
- 与课程组织者私信沟通。
- 借助智能小助手优化简历和了解系统流程。

### 4.2 课程组织者 MO

MO 是负责发布岗位、筛选候选人并安排面试/录用的教师或课程负责人。

核心诉求：

- 发布和维护课程助教、监考、活动等岗位。
- 查看岗位申请人和匹配度。
- 识别候选人的技能优势与短板。
- 对申请进行面试、录用、拒绝等处理。
- 与应聘者私信沟通。
- 管理自己发布的岗位状态。

### 4.3 管理员 Admin

管理员负责从全局视角查看系统录用结果和 TA 工作负荷。

核心诉求：

- 登录管理员后台。
- 查看 TA 已录用岗位数量、平均值、最高/最低值和负荷提示。
- 展开查看 TA 的具体录用岗位。
- 取消不合适的已录用分配，必要时让岗位重新开放。

### 4.4 访客

未登录用户可以访问首页、查看入口信息和浏览论坛/部分公开信息，但涉及申请、发帖、私信、工作台等操作时需要登录。

## 5. 当前已实现范围

### 5.1 多角色认证与个人中心

当前系统已实现 TA、MO、Admin 三类角色的注册、登录、退出和会话识别。

已实现能力：

- TA 注册、登录、注册确认页、退出。
- MO 注册、登录、退出。
- Admin 登录、注册、退出。
- `/personal-center` 作为统一个人中心入口，根据当前 session 自动跳转到对应角色页面。
- 登录状态通过 Servlet `HttpSession` 保存，TA、MO、Admin 分别使用独立 session 标识。
- 会话超时时间配置为 30 分钟。
- 登录页、工作台、论坛等页面设置 no-store 缓存策略，降低退出后通过浏览器后退看到敏感页面的风险。

### 5.2 TA 应聘工作台

TA 工作台是应聘者侧的核心页面，围绕资料、简历、岗位、申请和消息展开。

已实现能力：

- 查看个人资料。
- 编辑姓名、学号、技能标签等资料。
- 上传简历文件，支持 txt、pdf、doc、docx。
- 编辑和保存文本简历。
- 查看简历历史版本并恢复。
- 下载本人简历。
- 浏览岗位列表。
- 按关键词、岗位类型、技能等维度筛选岗位。
- 收藏岗位。
- 提交岗位申请并填写申请说明。
- 查看“我的申请”列表。
- 查看申请状态，包括待处理、面试、已录用、已拒绝、已取消。
- 对面试安排进行确认、拒绝或申请改期。
- 与 MO 进行站内私信沟通。

### 5.3 MO 招聘管理工作台

MO 工作台用于岗位发布、岗位维护、申请人筛选和消息沟通。

已实现能力：

- 查看自己发布的岗位。
- 发布新岗位。
- 编辑已有岗位。
- 关闭岗位。
- 查看岗位申请人列表。
- 基于技能匹配度查看候选人推荐。
- 查看候选人的技能命中和技能短板。
- 对候选人进行录用或拒绝。
- 安排面试时间和面试地点。
- 重新安排面试。
- 查看候选人统计信息，包括平均分、中位数和分数分布。
- 与 TA 进行站内私信沟通。

### 5.4 申请与面试流程

系统已形成从岗位发布到录用管理的基础闭环。

核心状态：

- `pending`：TA 已提交申请，等待 MO 处理。
- `interview`：MO 已安排面试。
- `accepted`：MO 已录用 TA。
- `rejected`：MO 已拒绝申请。
- `cancelled`：申请已取消或管理员取消录用。

主要流程：

1. MO 发布岗位。
2. TA 浏览岗位并提交申请。
3. MO 在申请人页查看候选人和匹配信息。
4. MO 可安排面试、录用或拒绝。
5. TA 对面试安排进行确认、拒绝或申请改期。
6. 录用结果进入管理员工作负荷统计。
7. 管理员可取消已录用记录，并根据逻辑重新开放岗位。

### 5.5 岗位与匹配推荐

岗位支持课程助教、监考、活动支持等类型。当前样例数据中包含软件工程、Web 开发实践、数据结构、机器学习、操作系统、数据库、监考、开放日志愿者、毕业设计档案整理等岗位。

已实现能力：

- 岗位标题、描述、类型、所需技能、状态和创建时间。
- open / closed 岗位状态。
- 基于 TA 技能、简历文本和岗位所需技能的规则匹配。
- 匹配分展示。
- 技能短板和优势提示。
- 可选开启 embeddings 语义匹配，用向量相似度替代或增强规则打分。
- embeddings 结果缓存到 JSON 文件，减少重复调用外部服务。

### 5.6 管理员工作负荷管理

管理员模块当前聚焦于录用结果和负荷均衡。

已实现能力：

- 管理员登录和注册。
- 查看所有已录用申请形成的 TA 工作负荷。
- 查看录用总量、平均值、最高值、最低值。
- 标记相对偏高或偏低的工作负荷。
- 展开查看某个 TA 的具体录用岗位。
- 取消已录用申请。

当前 README 中提到“转移录用”，但代码探索结果显示当前实际能力更接近“取消已录用分配并在必要时重新开放岗位”。PRD 后续版本如需“转移录用”，应将其作为新增需求明确设计。

### 5.7 论坛与社交沟通

系统已实现基础论坛和站内沟通能力。

论坛已实现能力：

- 浏览帖子列表。
- 查看帖子详情。
- 搜索帖子。
- 排序和分页。
- 登录用户发帖。
- 登录用户回帖。
- 支持 TA、MO、Admin 三类登录用户参与。

私信与好友已实现能力：

- TA 与 MO 可基于申请关系或好友关系进行私信。
- 会话列表展示。
- 发送消息。
- 未读状态和已读状态记录。
- 数据文件中存在好友请求和好友关系模型。

当前页面层未看到独立、完整的好友管理入口，因此好友能力更适合作为“已具备数据模型和部分支撑能力”，而不是完整用户可见模块。

### 5.8 智能小助手

系统已接入可选智能小助手，面向 TA 提供招聘、简历和系统使用相关帮助。

已实现能力：

- `/assistant` 小助手页面。
- 调用 OpenAI 兼容 Chat Completions 接口。
- 支持 Kimi、Qwen、OpenAI 等提供方配置。
- 服务端对提问范围做限制，明显无关问题会拒答。
- 支持粘贴简历文本。
- 支持上传简历文件并抽取文本。
- 支持使用已保存简历作为上下文。
- 查询额度。
- 每月免费额度。
- 免费额度用完后扣付费点数。
- 兑换码充值。
- 配置齐全时支持微信 Native 扫码下单、订单查询和支付回调入账。

注意：当前配置文件中存在明文 API Key 示例/实值。产品交付或公开仓库使用时，应迁移到环境变量或外部私密配置，并对历史提交进行密钥轮换。

### 5.9 多语言

系统已具备基础国际化能力。

已实现能力：

- `LocaleFilter` 支持语言切换。
- 支持 `zh_CN` 和 `en` 文案资源。
- 可通过请求参数和 cookie 记住语言偏好。

## 6. 核心用户流程

### 6.1 TA 申请岗位流程

1. TA 从首页进入 TA 登录/注册。
2. TA 完成注册并登录。
3. TA 进入工作台，补充个人资料和技能标签。
4. TA 上传或编辑简历。
5. TA 浏览岗位，使用筛选条件找到目标岗位。
6. TA 收藏岗位或直接提交申请。
7. TA 在“我的申请”中查看申请状态。
8. 若 MO 安排面试，TA 确认、拒绝或申请改期。
9. TA 与 MO 通过站内信补充沟通。
10. TA 查看最终录用或拒绝结果。

### 6.2 MO 发布与筛选流程

1. MO 从首页进入 MO 登录/注册。
2. MO 登录后进入工作台。
3. MO 发布岗位，填写标题、描述、类型和所需技能。
4. MO 查看自己发布的岗位列表。
5. MO 进入岗位申请人页面。
6. MO 根据匹配分、技能短板、申请说明和简历信息筛选候选人。
7. MO 安排面试，或直接录用/拒绝。
8. MO 根据 TA 的面试反馈重新安排面试或调整处理结果。
9. MO 通过站内信和 TA 沟通。

### 6.3 管理员查看负荷流程

1. Admin 进入 `/admin/auth` 登录。
2. Admin 进入 `/admin/workload`。
3. Admin 查看整体录用数量和负荷分布。
4. Admin 展开查看某个 TA 的具体录用岗位。
5. Admin 对不合理录用进行取消处理。

### 6.4 AI 小助手使用流程

1. TA 登录后进入 `/assistant`。
2. TA 选择模型或使用默认模型。
3. TA 输入招聘、简历、申请流程相关问题。
4. TA 可粘贴简历、上传简历文件或使用系统已保存简历。
5. 系统检查额度。
6. 额度充足时调用外部大模型并返回答案。
7. 免费额度不足时扣付费点数。
8. 额度不足时返回充值提示，可使用兑换码或微信支付充值。

## 8. 非功能需求

### 8.1 性能

当前系统使用 JSON 文件存储，适合小规模演示和课程项目。建议目标规模为几十到数百条岗位/申请数据。若数据量扩大，需要迁移到数据库并增加索引、分页和事务。

### 8.2 安全

当前已具备基础登录会话和缓存控制，但生产安全不足。

需要关注：

- 密码当前使用 SHA-256 哈希，不含盐和慢哈希，生产应改为 BCrypt、Argon2 或 PBKDF2。
- 数据文件为明文 JSON，应限制服务器文件权限。
- API Key 不应写入仓库，应改为环境变量或外部配置。
- AI 和 embeddings 可能向第三方发送简历和岗位文本，需要用户授权和隐私提示。
- 支付回调必须使用 HTTPS，并严格校验微信签名。

### 8.3 可用性

当前通过 JSP 页面实现主要功能，适合桌面浏览器使用。后续可增强：

- 移动端适配。
- 表单错误提示一致性。
- 空状态、加载状态和成功反馈。
- 更清晰的申请状态时间线。

### 8.4 可维护性

当前采用 Servlet + Service + Model + Storage 的轻量结构，便于课程理解。随着功能增加，建议逐步引入：

- 数据库和 DAO/Repository 层。
- 统一权限校验。
- 统一 API 响应结构。
- 自动化测试。
- 配置分环境管理。

## 9. 数据与存储

当前应用启动后将数据目录设置为 Web 应用根目录下的 `data/`，使用 Gson 读写 JSON 文件。

主要数据对象：

- `applicants.json`：TA 用户。
- `module_organisers.json`：MO 用户。
- `admins.json`：管理员。
- `jobs.json`：岗位。
- `applications.json`：申请、面试和录用状态。
- `messages.json`：站内私信。
- `dm_read_states.json`：私信已读状态。
- `forum_threads.json`：论坛帖子。
- `forum_replies.json`：论坛回复。
- `friend_links.json`：好友关系。
- `friend_requests.json`：好友请求。
- `embeddings.json`：语义匹配向量缓存。
- `assistant_usage.json`：小助手额度使用。
- `assistant_pay_orders.json`：小助手支付订单。
- `data/resumes/`：简历文本和上传文件。

## 10. 技术实现概览

当前技术栈：

- 后端语言：Java 11。
- Web 技术：Servlet 4.0、JSP、JSTL。
- 构建工具：Maven。
- 打包方式：WAR。
- 本地运行：Tomcat Maven Plugin，默认端口 8082，默认上下文路径 `/`。
- JSON：Gson。
- 简历解析：Apache PDFBox、Apache POI。
- 外部 AI 调用：Java 11 `HttpClient` 调用 OpenAI 兼容接口。
- 数据存储：Web 应用目录下 JSON 文件和简历文件。

主要入口：

- `/`：首页。
- `/personal-center`：个人中心网关。
- `/ta/auth`、`/ta/dashboard`、`/ta/profile`：TA 认证、工作台和个人资料。
- `/mo/auth`、`/mo/dashboard`、`/mo/profile`、`/mo/job-applicants`：MO 认证、工作台、个人资料和申请人管理。
- `/admin/auth`、`/admin/workload`：管理员认证和工作负荷。
- `/forum`：论坛。
- `/assistant`：智能小助手页面。
- `/api/assistant/chat`：小助手对话接口。
- `/api/assistant/extract-resume`：简历文件抽取接口。
- `/api/assistant/quota`：额度查询接口。
- `/api/assistant/quota/topup`：额度充值码接口。
- `/api/assistant/pay/wechat/native`：微信 Native 下单接口。
- `/api/assistant/pay/wechat/order`：微信订单查询接口。
- `/api/assistant/pay/wechat/notify`：微信支付回调接口。

## 12. 当前风险与限制

- 文件存储不适合高并发和大数据量场景。
- JSON 明文数据缺少数据库级事务、备份、迁移和审计。
- 密码哈希方式不适合生产环境。
- 配置文件中存在明文密钥风险。
- AI 能力依赖第三方 API 可用性、额度和网络。
- 简历文本发送给第三方模型存在隐私合规风险。
- 微信支付需要公网 HTTPS、商户配置和严格验签，开发环境可能无法完整闭环。
- 自动化测试覆盖不足。
- 好友、课程管理、日历和通知等能力还不完整。
